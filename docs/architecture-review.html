S1 · Strong Security + Performance 双修 验证成本极低

CSP seam 命名偏离 + RootLayout 把全站拖入 dynamic

涉及文件：src/proxy.ts · src/app/layout.tsx · next.config.ts
Problem

CSP nonce 的 adapter 暴露在 src/proxy.ts 的 proxy() 上，但 Next.js 约定的 seam 是 src/middleware.ts 的 middleware()。 全仓 grep middleware 仅命中 proxy.ts 内部一条注释；src/middleware.ts 不存在。

若 Next 16.2 不识别 proxy.ts，则：x-nonce 永不被设置 → layout.tsx 的 headers().get('x-nonce') 永远 null → 内联主题脚本无 nonce 渲染 → 由于 next.config 的 securityHeaders 也不含 CSP，生产站点实际无任何 CSP。

与此同时，RootLayout 是 async 且调用 headers()，这会让所有子路由继承 dynamic rendering，瓦解 AGENTS.md 宣称的 SSG。

Solution（两条路）

路线 A — 接受 dynamic：把 proxy.ts 重命名为 middleware.ts + middleware()，显式 export const dynamic = 'force-dynamic'。保留 per-request nonce，但放弃整站 SSG。

路线 B — 恢复 SSG（推荐）：外移主题探测脚本到 /theme-init.js 静态资产，CSP 改用 script-src 'self'（无需 nonce），layout 恢复 static。next.config 加静态 CSP header。

第一步（零成本验证）：跑一次 next build && next start，curl 任意页面看 response header 是否含 Content-Security-Policy。

posts.ts god module 拆分 + zod schema 单一来源

涉及文件：src/lib/posts.ts (255 行 / 13 export) · src/types/index.ts · scripts/generate-rss.ts · scripts/check-seo.ts
Problem

posts.ts 单文件承载 5 类职责：
(a) frontmatter zod schema 定义与校验
(b) MDX 正文清洗（stripMdxForSearch / extractPostExcerpt / buildPostSearchText / extractPostHeadings）
(c) reading-time 副作用集成
(d) 缓存生命周期管理（_cache + isVisible 环境判定）
(e) 查询/筛选/排序/分页（getPostsByTag / getAdjacentPosts / getRelatedPosts / getSeriesPosts / getPaginatedPosts）

同时 zod schema 与 TS interface 双源：schema 里 published 是 .optional().default(true)，types 里是 boolean（必填）— 已出现语义漂移。

RSS/check-seo 各自重写 schema 子集：generate-rss.ts 只校验 5 字段；check-seo.ts 用 9 个 if 分支重做校验。posts.ts 加新必填字段时，RSS 静默接受缺字段文件，check-seo 不会报错。

Solution

拆分为 3 层 + 1 个 schema source：

src/lib/posts/schema.ts — 唯一 zod schema + export type PostFrontmatter = z.infer<typeof postFrontmatterSchema> 派生类型
src/lib/posts/repository.ts — 缓存 + 读写（getAllPosts / getPostBySlug）
src/lib/posts/query.ts — 筛选/排序/分页（getPostsByTag / getAdjacentPosts / getRelatedPosts / getSeriesPosts / getPaginatedPosts）
src/lib/posts/search-text.ts — MDX 清洗（stripMdxForSearch / extractPostExcerpt / buildPostSearchText / extractPostHeadings）

RSS/check-seo 直接 import 同一 schema，不再各自重写子集。
src/types/index.ts 从 zod 派生类型，删除手写 interface。

顺手解决：F8 zod/TS 双源、scripts #2 schema 复制、lib F6 排序模式重复（提炼 comparePostsByDate）、lib F12 isVisible 直接读 env。

S3 · Strong CI 时间 -50% 性能回归守门

CI build artifact 共享 + Lighthouse 接入 deploy gate

涉及文件：.github/workflows/ci.yml
Problem

5 个 job 各自 build：quality / bundle-analyze / e2e / lighthouse / deploy 每个都跑 pnpm install --frozen-lockfile + pnpm build。
bundle-analyze 与 quality 的差别只是 ANALYZE=true 环境变量，却完全重跑。

Lighthouse 不是 deploy gate：deploy 的 needs 是 [quality, e2e]，lighthouse 失败不阻断生产部署。方向 B 的工作只完成一半 — 阈值升级到 error，但 deploy 路径绕过它。性能回归可悄悄进生产。

bundle-analyze 是孤儿 job：它失败不影响任何 job，唯一产物是 14 天保留的 artifact，没人会主动看。

Solution

三步改造：

(1) 新增 build job 作为公共依赖：跑 pnpm build 一次，actions/upload-artifact@v4 上传 .next/。
(2) quality / e2e / lighthouse / deploy 都 needs: [build]，用 actions/download-artifact@v4 拉取构建产物（不再各自 build）。
(3) deploy 改为 needs: [quality, e2e, lighthouse]，让性能回归也阻断生产。

bundle-analyze 保留 needs: [build]，但用 ANALYZE=true 重新构建（因为需要 analyzer 输出，不能用普通 build artifact）。

S4 · Strong 解耦内容编辑与代码测试反馈循环

ContentSource seam 显式化 + cache 可重置

涉及文件：src/lib/cache.ts · src/lib/content-source.ts · src/lib/posts.ts · src/lib/projects.ts · src/lib/posts.test.ts
Problem

Cache 隐式 import 全局单例：cache.ts 的 getOrCompute 直接 import { getContentSource } 调 source.readDir / source.getMtime 判断失效。ContentSource 的 _activeSource 是 service locator 全局态。

setContentSource 不通知 cache：切换 source 时 cache 持有旧 source 的过期数据。

_cache 未导出：posts.test.ts 调真实 getAllPosts 后无法直接 invalidate，必须靠 mtime 变化间接触发。测试间污染风险高。

测试读真实 fs：posts.test.ts 断言 getPostBySlug('go-cli-tool').title contains 'Go'、getSeriesPosts('vps-initial-setup') 返回固定 5 篇 — 内容改动会让 6+ 测试断裂，把内容编辑和代码测试的反馈循环耦合在一起。

Solution

显式注入：createCache<T>({ source, watchPath }) 把 ContentSource 作为构造参数注入，删除 cache.ts 对 getContentSource 的隐式 import。

cache 可重置：导出 invalidatePostsCache() / invalidateProjectsCache() 单行函数，或注册全局 resetAllCaches()。

source 切换触发失效：setContentSource(newSource) 内部调用 resetAllCaches()，确保 source 变更后 cache 必失效。

posts.test.ts 用 in-memory source：把 schema-related 断言改为 setContentSource(inMemorySource({fixtures})) 注入夹具；保留 1-2 个真实 fs 集成测试。

顺手解决：lib F1/F3/F11、scripts #4 ContentSource 未利用。

S5 · Strong 9 个未测试组件退化为可测 无障碍审计收敛

客户端 hooks 提炼（reduced-motion + persisted-enum + in-view）

涉及文件：ui/ParticleCanvas.tsx · ui/MagneticCard.tsx · ui/BackToTop.tsx · home/EditorialHero.tsx · home/RevealOnScroll.tsx · blog/ReadingPreferences.tsx · ui/ThemeToggle.tsx · blog/TableOfContents.tsx · comments/Giscus.tsx
Problem

prefers-reduced-motion 在 4 处重复（ParticleCanvas / MagneticCard / EditorialHero / RevealOnScroll），且降级策略不一致 — 一个跳出 effect、一个跳过 transform、一个立即添加 is-visible、一个直接 return。

localStorage + 枚举循环在 2 处重复（ThemeToggle 3 态 theme / ReadingPreferences 3 态 fontSize + 3 态 width），共享完全相同的形状但 interface 是隐式的。

IntersectionObserver 在 4 处重复（RevealOnScroll / TableOfContents / Giscus / ParticleCanvas），4 套样板 + 4 种 rootMargin/threshold。

横切问题：9/13 个 client 组件无单元测试，恰好是浏览器 API 密集的那批。当前测试面是"容易测的面"，不是"实际出 bug 的面"。

Solution

提炼 3 个薄 hook：

usePrefersReducedMotion(): boolean
usePersistedEnum<T>(key, defaultValue, validValues): [T, next: () => void]
useInView(ref, options): boolean（options: { threshold, rootMargin, once }）

9 个未测试组件退化为 "render hook → assert return"，jsdom 不再需要 mock 全套浏览器 API。

风险：useInView 边缘浅（deletion test），价值在于集中 "unmount 时 disconnect" 契约而非代码缩减。需要一次 interface 设计对话确认 hook 形状。

动态路由 adapter + listing metadata 一致化

涉及文件：src/app/blog/[slug]/page.tsx · src/app/projects/[id]/page.tsx · src/app/tags/[tag]/page.tsx · src/app/categories/[category]/page.tsx · src/app/{blog,projects,tags,categories,about,links}/page.tsx
Problem

4 条动态路由独立实现同构的 3 段式 interface：
(a) generateStaticParams 调 getAllXSlugs
(b) generateMetadata 调 getXBySlug + 拼 canonical/title
(c) default page 调 getXBySlug + notFound() 守卫

形状一致但已 drift：
• canonical URL 编码不一致（tags 用 encodeURIComponent(tagSlug)、categories 用 encodeURIComponent(categoryName)、projects 完全省略 canonical、blog 用裸 slug）
• title 模板不一致（blog 用 title: post.title 依赖 root template，其余用 ${name} | ${SITE_CONFIG.name}）
• notFound() 守卫位置不一致（blog/projects 在 generateMetadata、tags/categories 在组件内）

6 个 listing 页无 metadata：/blog /projects /tags /categories /about /links 全部依赖 root layout 的 title.default，浏览器 tab 和 OG title 显示裸 "Site Name"。

createDynamicRoute adapter：
createDynamicRoute({ getAllSlugs, getBySlug, buildMetadata, render })
返回 { generateStaticParams, generateMetadata, default: Page } 三件套。adapter 内集中 canonical 编码、title 模板、notFound 守卫。

listingMetadata helper：
listingMetadata(title, description, path): Metadata
6 个 listing 页一行 import。

顺带解决 App #4：删除 blog/[slug]/opengraph-image.tsx 的 generateStaticParams（让父路由供给 params）。

顺带解决 App #6：所有 listing 页补 metadata。
