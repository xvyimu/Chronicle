# 博客项目优化目标与实施方案

状态：Phase 1-3 + P2 UX polish implemented
日期：2026-07-06  
适用范围：架构优化、用户体验优化、视觉设计优化  
执行原则：先文档、再分阶段小步修改、每阶段验证。

## 1. 项目现状分析

### 1.1 当前定位

当前项目是一个本地内容驱动的个人博客与作品集站点，核心能力包括：

- 博客文章：`content/blog/*.mdx`
- 关于页：`content/about.mdx`
- 作品集：`data/projects.json`
- 个人收藏：`data/links.json`
- 页面渲染：Next.js App Router
- 内容解析：MDX、frontmatter parser、Zod schema、reading-time、Shiki
- 搜索：`fuse.js` 客户端模糊搜索
- 质量门禁：Vitest、Playwright、ESLint、TypeScript、SEO 检查、bundle budget、Lighthouse CI、Vercel 部署

### 1.2 技术架构

| 层级   | 当前实现                                                                   | 评价                                                                   |
| ------ | -------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| 框架层 | Next.js 16.2 App Router、React 19.2                                        | 已具备现代路由、metadata、OG、manifest、sitemap 能力                   |
| 内容层 | MDX + JSON + `ContentSource` + repository                                  | 边界清晰，测试可注入内存源                                             |
| 查询层 | `lib/posts/query.ts` 等纯函数                                              | 易测，适合继续扩展                                                     |
| 缓存层 | `createCache<T>`                                                           | 已有统一缓存入口，未来可迁移到更细粒度策略                             |
| UI 层  | `components/home`、`components/blog`、`components/layout`、`components/ui` | 基本分层明确，仍有部分组件样式与行为可收口                             |
| CSS 层 | Tailwind v4 + 显式 CSS module import + BEM                                 | 已规避 Tailwind v4 `@import` 限制，但 import 顺序与 token 重复需要治理 |
| 安全层 | `proxy.ts` CSP nonce + next.config security headers                        | 生产边界成熟，但会导致动态渲染，需要继续接受该取舍                     |
| 部署层 | GitHub Actions + Vercel + production smoke                                 | 完整，适合作为优化验收基线                                             |

### 1.3 当前已有优势

- 内容仓储已经抽象出 `ContentSource`，不是直接把 fs 散落在组件中。
- 动态路由已有 `createDynamicRoute`，减少了 `[slug]`、`[tag]`、`[category]` 类页面重复。
- 首页、文章页、链接页、归档页已经有较完整的功能面。
- 文章页已有目录、阅读进度、阅读设置、相关文章、上一篇/下一篇、Giscus。
- 搜索已经动态加载 `fuse.js`，避免直接扩大首屏 bundle。
- CI 已覆盖格式、单测、类型、构建、bundle、E2E、Lighthouse 与部署。

## 2. 优化目标

### 2.1 总体目标

本次优化不是重写项目，而是在不破坏现有功能的前提下，提升四项能力：

1. 加载速度更稳：守住 bundle、Lighthouse 和真实用户指标。
2. 内容可读性更强：让文章、代码块、目录、推荐和移动端阅读更顺。
3. 访问路径更清楚：让首页、博客、分类、专题、收藏之间形成清晰探索路径。
4. 工程结构更可维护：减少分散配置、重复 token、重复 UI 状态和隐性约定。

## 3. 架构优化方案

### 3.1 方案 A：渐进式收口现有架构

保留当前 App Router、MDX、JSON、CSS modules、CSP nonce 和 Vercel 部署链路，只在现有边界上做集中配置、公共逻辑抽取、样式 token 收敛和测试补强。

### 3.2 方案 B：引入内容构建索引层

在构建阶段生成文章索引、搜索索引、图片清单和内容关系图，页面只消费生成产物。适合文章量较大或需要更强内容治理时使用。

### 3.3 方案 C：迁移外部 CMS 或数据库

将 MDX/JSON 迁移到外部内容源，保留 `ContentSource` 抽象作为适配层。适合多人编辑、后台管理或动态内容需求。

### 3.4 取舍矩阵

| 维度       | 方案 A：渐进式收口                   | 方案 B：构建索引层             | 方案 C：外部 CMS/DB                    |
| ---------- | ------------------------------------ | ------------------------------ | -------------------------------------- |
| 实现复杂度 | 低到中，主要是本地重构和测试         | 中，需要新增生成脚本和产物校验 | 高，需要认证、数据迁移、部署变量和备份 |
| 运维成本   | 低，沿用当前 Vercel + GitHub Actions | 中，CI 需生成并检查索引        | 高，新增外部服务和失败模式             |
| 性能收益   | 稳定，减少客户端负载和重复逻辑       | 高，适合大内容量搜索和关联推荐 | 不确定，取决于缓存和数据源延迟         |
| 风险       | 低，可小步回滚                       | 中，生成产物可能和源码不同步   | 高，内容迁移和生产数据风险大           |
| 团队熟悉度 | 高，完全贴合当前项目                 | 中，需要维护构建脚本           | 低到中，需要新后台流程                 |
| 演进空间   | 足够支撑近期优化                     | 支撑内容规模增长               | 支撑多人运营                           |

推荐：先执行方案 A。  
理由：当前项目内容规模和架构成熟度不需要重平台化，最大收益来自收口、体验和视觉一致性。方案 B 作为文章量增长后的演进路径，方案 C 暂不建议执行。

## 5. 视觉设计优化方案

### 5.1 当前视觉风格

当前风格可以概括为：低饱和纸张背景、蓝灰/鼠尾草强调色、细线分隔、Editorial hero、大标题衬线字体、暗色模式自动适配。

这是适合个人技术博客的方向，应继续保留：

- 简洁、克制、留白充足。
- 有艺术感，但不把阅读内容压过去。
- 卡片半径控制在小到中等，不做过度圆角和浮夸阴影。
- 首页可以更有品牌感，内页保持阅读优先。

### 5.2 设计规范目标

| 项目     | 规范方向                                                              |
| -------- | --------------------------------------------------------------------- |
| 主色     | `--brand` 鼠尾草蓝灰，用于链接、强调、焦点、当前状态                  |
| 辅助色   | `--featured` 暖金棕，只用于精选、特殊状态，不做大面积背景             |
| 背景色   | `--bg`、`--bg-soft`、`--surface` 三层足够，避免新增相近色             |
| 文本色   | `--text`、`--text-soft`、`--text-dim` 三层语义                        |
| 字体     | 正文 Noto Sans SC，代码 JetBrains Mono，展示 Cormorant Garamond       |
| 字号层级 | hero/title/section/card/body/caption 分层，内页标题不滥用 hero 级尺寸 |
| 行高     | 正文 1.75-1.85，卡片摘要 1.55-1.7，按钮 1                             |
| 圆角     | 工具控件 6-8px，普通卡片 8-14px，pill 状态用 full                     |
| 阴影     | 默认细线优先，阴影只在浮层和 hover 强调中使用                         |
| 间距     | 使用 `--space-*`，section 以 48/64/72px 为主要节奏                    |
| 图标     | 优先使用一致 stroke 风格；未来可统一到 icon 组件                      |

## 9. 风险评估

| 风险              | 等级 | 触发场景                                 | 缓解措施                               | 回滚方案                          |
| ----------------- | ---- | ---------------------------------------- | -------------------------------------- | --------------------------------- |

| 动态渲染/CSP 回归 | 中   | 修改 layout、JSON-LD、script、proxy      | 不改变 nonce 读取链路，测试生产构建    | 回滚 layout/proxy 相关改动        |
| 搜索交互回归      | 中   | 修改 SearchBar、Fuse、Header 搜索入口    | 保留现有 API，补键盘测试               | 回滚 SearchBar/Header 改动        |

| Bundle 增大       | 中   | 新增依赖、图标库、搜索能力               | 优先不新增依赖，动态加载，跑 budget    | 移除新增依赖或拆懒加载            |

## 10. 回滚策略

1. 每阶段保持小范围修改，避免一次性跨越架构、UX 和视觉。
2. 每个任务完成后先运行对应最小验证，再进入下一任务。
3. CSS 改动优先保留旧变量别名，不直接删除旧 class。
4. 组件 API 改动优先兼容旧 props，再在后续阶段清理。
5. 如果阶段验证失败，优先回滚该阶段最后一个任务，而不是重置整个项目。
6. 生产部署前必须确认 `git status`、CI、构建、E2E 和生产内容检查。
