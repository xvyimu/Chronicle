# 提示词优化交付概览

## 完成内容

将原始提示词“深度研究项目，推进项目”优化为一份可直接交给代码助手执行的中文提示词，并完成产品需求复核。

## 关键优化

- 补齐项目目标、里程碑、范围限制与验收标准。
- 建立“基线分析 → 差距评估 → 优先级计划 → 逐步实施 → 持续验证 → 交付汇报”闭环。
- 要求结论以仓库内容或运行结果为依据，并区分事实、推断和待确认项。
- 不预设具体技术栈，避免无依据重构或引入依赖。
- 最终版本共 667 个字符，低于 800 字限制。

## 最终增强提示词

> 请深度研究当前项目，并将其推进到一个可验证的下一里程碑。
> 项目目标：【填写】；期望里程碑/交付物：【填写】；范围与限制：【填写】；验收标准：【填写】。若这些信息未提供，先从仓库、文档和任务记录中寻找依据，再集中提出最少量的关键问题；不影响安全推进的事项无需等待。
>
> 1. 建立基线：阅读项目结构、文档、配置、依赖和核心代码，识别实际架构、技术栈、运行方式及当前阶段；运行已有构建、检查和测试，记录原有失败。所有结论区分“事实、推断、待确认”，不得臆测。
> 2. 评估差距：围绕项目目标检查架构适配性、代码质量、未完成功能与缺陷、性能瓶颈和技术难点、异常处理与日志、单元/集成测试；同时关注相关的安全、配置、依赖和数据兼容风险。只报告有代码或运行结果支持的问题。
> 3. 制定计划：按P0/P1/P2列出任务，说明用户价值、依据、依赖、风险、预计影响范围及可验证的完成标准。优先选择能推动里程碑且风险可控的最小闭环。
> 4. 逐步实施：若无关键阻塞，不要停留在分析，直接完成最高优先级事项。沿用现有技术方案和代码规范，保持小步、可回退；未经论证不做大规模重构、不替换技术栈、不引入多余依赖。遇到重大歧义、破坏性操作或权限问题时先说明并请求确认。
> 5. 持续验证：每步修改后执行相关测试、静态检查和构建；为新增或修复行为补充必要测试，并进行关键路径回归。失败时定位原因，修复或明确记录阻塞，不得将未验证结果称为完成。
> 6. 交付汇报：总结已完成内容、修改文件、验证命令与结果、遗留问题、风险及下一步建议；明确区分已完成、部分完成、未开始和受阻项。

## 后续建议

正式使用前，优先填写“项目目标、期望里程碑、范围与限制、验收标准”四个字段；若暂时未知，也可让代码助手先从仓库中建立事实基线。
