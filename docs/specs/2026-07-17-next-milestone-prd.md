- **Language**：中文
- **Programming Language**：Next.js 16.2 App Router、React 19.2、TypeScript 5 strict、Tailwind CSS 4；工具脚本使用 Node.js ESM
- **Project Name**：`documentation_link_ci_gate`
- **原始需求复述**：基于仓库当前事实，识别一个风险可控、可在本轮实现并验证的下一里程碑；不得更换技术栈或进行大规模重构。

## 2. 当前基线

### 已确认事实

- 产品是 MDX/JSON 驱动的个人博客兼作品集，生产域名为 `https://incca.ccwu.cc`；首页、博客、分类/标签/专题、项目、收藏、搜索、Feed、SEO、安全头与评论等核心能力已落地。
- 根 `TODO.md` 声明工程侧 P0-P10 已完成，剩余项均依赖账号、真实流量、内容规模或图片素材，当前不能无条件推进。
- 最新记录的生产基线为提交 `8ee2e71`、CI run `29573545749` 全绿；本地记录为 Vitest 77 文件/597 用例、Playwright 5 文件/48 用例。
- 当前工作树已有大量未提交的文档收尾改动，并新增 `scripts/check-doc-links.mjs`、其 5 个 Node fixture 测试和 Vitest 桥接测试；不得覆盖或清理这些改动。
- `node scripts/check-doc-links.mjs` 当前实测通过，扫描 54 个 Markdown 文件。
- 链接检查器已写入 README 和维护流程，但 `package.json` 没有对应脚本，`.github/workflows/ci.yml` 也没有执行仓库链接扫描；现有 Vitest 桥接只运行解析器 fixture，不验证仓库当前文档链接。

### 推断

- 文档已成为内容发布、架构边界和上线操作的入口；失效内部链接会增加接手和发布误操作概率。
- 将现有检查器接入标准命令和 CI，变更面小、无需 UI 改动，且能形成“发现坏链—阻止合并—修复复验”的最小闭环。

### 待确认

- 上述未提交文档改动是否计划与本里程碑同批提交；本轮只能保留并增量修改。
- CI 命令最终命名采用 `check:docs` 还是 `check:doc-links`；推荐前者，便于未来承载其他文档校验。

## 3. 差距与优先级

- **P0（上线阻断）**：未发现可由仓库证据确认的新 P0；不得把账号/流量依赖伪装成工程阻断。
- **P1（本轮推荐）**：文档链接扫描未进入标准 package script 与 CI，真实仓库坏链仍可能在质量流水线中漏过。
- **P2（暂缓）**：移动端 Lighthouse FCP/LCP 警告需独立重复基线或真实用户数据后再立性能专项；GSC/Bing、Speed Insights、外部搜索、正文图 LQIP、Cache Components 与 CSS 深度下沉均未满足触发条件。

## 4. 产品定义

### Product Goals

1. **自动防回归**：每次 push/PR 的 quality job 必须扫描维护范围内的内部 Markdown 链接，坏链使流水线非零退出。
2. **统一开发入口**：维护者可通过一个 `pnpm` 命令本地执行与 CI 完全相同的链接检查。
3. **保持低风险**：复用现有无依赖脚本与 fixture，不改变页面、内容模型、运行时或部署架构。

### 非目标

- 不校验外部 URL 的在线可达性，不引入网络请求或第三方链接服务。
- 不改变 Markdown 内容、路由、视觉或用户交互。
- 不把历史 spec 的正文格式化或历史状态改写为当前事实。
- 不处理需要账号、生产流量、未来内容规模或素材的 TODO。

### User Stories

- As a 文档维护者, I want 在本地用标准命令检查内部链接, so that 我能在提交前发现路径错误。
- As a 代码审查者, I want PR 自动阻断坏链, so that 合并后的接手与发布文档保持可导航。
- As a 后续 Agent, I want README、CI 和实际命令一致, so that 我无需猜测文档验证步骤。

## 5. Requirements Pool

### P0 — Must have（本里程碑闭环）

1. `package.json` 必须提供标准文档检查命令，执行现有仓库链接扫描；命令在任意坏链存在时必须返回非零退出码。
2. GitHub Actions `quality` job 必须显式运行该命令，且位于依赖安装后、构建/部署前。
3. 必须保留并通过现有 5 个解析 fixture，覆盖括号路径、引用定义、代码块忽略、缺失目标与非法 URI。
4. 必须增加或保留一项仓库级验证，证明当前根文档与 `docs/**` 实际扫描通过，而不只验证 parser fixture。
5. README/AGENTS/接手指南中的命令名称与 CI 行为必须一致，不得同时保留多个互相冲突的入口。

### P1 — Should have

1. CI 日志应输出扫描文件数；失败日志应至少包含来源文件、目标和失败原因。
2. 检查范围应继续覆盖根维护文档和 `docs/**`，并忽略 fenced/inline/indented code、纯锚点和外部协议链接。
3. 新增行为应有测试证明“有效仓库通过、注入坏链失败”。

### P2 — Nice to have

1. 后续可评估锚点存在性校验，但需单独处理 GitHub/Markdown 标题 slug 兼容，不纳入本轮。
2. 后续可评估外链巡检或定时任务，但不得阻塞当前 PR 的确定性 CI。

## 6. 验收标准

1. 在未改坏文档时，标准命令退出码为 0，并报告扫描文件数大于 0。
2. 在测试 fixture 中加入指向不存在文件的内部链接时，命令退出码非 0，输出包含来源、目标和 `target does not exist`；fixture 清理后恢复通过。
3. `pnpm test` 通过，且链接解析器 5 个 fixture 全绿；仓库级扫描测试/命令也通过。
4. `pnpm format:check`、`pnpm format:docs:check`、`pnpm lint`、`pnpm typecheck` 与 `git diff --check` 通过。
5. CI 配置可静态确认 quality job 调用标准命令；本轮若不获授权，不要求实际 push、部署或生产验证。
6. 页面 DOM、样式、路由清单、构建产物和生产运行时行为无变化。

## 7. UI / 交互影响

- **访客端**：无 UI、布局、交互和可访问性变化。
- **开发者端**：新增统一 CLI 入口；CI 出现明确的文档链接门禁日志与失败信息。

## 8. 风险与控制

- **误报历史示例链接**：沿用代码块与外部链接忽略规则，并用现有 fixture 防回归。
- **扫描范围扩大导致 CI 失败**：先以当前 54 个 Markdown 文件通过结果为基线；失败必须修正真实坏链或补充有测试依据的解析规则，禁止粗暴排除整个目录。
- **覆盖未提交工作**：实现时只做增量编辑，修改前读取当前文件内容，不清理工作树。
- **重复门禁增加耗时**：脚本无网络、无第三方依赖，预期秒级；仍应在 CI 记录实际耗时。

## 9. 推荐里程碑

**M11：文档链接完整性自动门禁**

交付顺序：统一 `pnpm` 命令 → 补仓库级成功/失败验证 → 接入 quality job → 同步维护文档 → 运行文档、类型、lint、测试与 diff 门禁。完成定义是坏链可被本地和 CI 同一命令稳定阻断；不包含 push、部署和任何产品页面改动。
